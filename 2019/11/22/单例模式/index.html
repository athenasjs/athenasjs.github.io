单例模式

单例模式的核心是保证一个类只有一个实例，并且提供一个访问实例的全局访问点。

单例的使用场景

主要用在那些只需要一个对象的场合，以及避免程序行为异常，资源使用过量或者结果不一致，具体场景有：

  1. Spring中bean对象的实现方式
  2. servlet中每个servlet的实例
  3. Spring MVC框架中，控制器对象是单例模式
  4. 应用程序的日志应用，一般都用单例模式实现，这一般是由于共享的日志文件一直处于打开状态，因为只能有一个实例去操作，否则内容不好追加。
  5. 操作系统的文件系统，一个操作系统只能有一个文件系统。
  6. 项目中读取配置文件的类，一般也只有一个对象，没有必要每次使用配置文件数据，都new一个对象去读取。
  7. 数据库连接池的设计一般也是采用单例模式，因为数据库连接是一种数据库资源。

    单例模式优点

  8. 由于单例模式只生成一个实例，减少系统性能开销，当一个对象的产生需要比较多的资源时，比如读取配置、产生其他依赖对象时，则可以通过在应用启动时直接产生一个单例对象，然后永久驻留内存的方式来解决。
  9. 单例模式可以在系统设置全局的访问点，优化共享资源访问，例如可以设计一个单例类，负责所有数据表的映射处理。

单例的实现方式

常见的单例模式实现方式有：

  1. 饿汉式： 线程安全，调用效率高，但是不能延迟加载
  2. 懒汉式： 线程安全，调用效率不高，能延迟加载
  3. 双重检测锁模式： 由于JVM底层内部模型原因，偶尔会出问题
  4. 静态内部类方式：线程安全，资源利用率高，可以延时加载
  5. 枚举单例： 线程安全，调用效率高，但是不能延时加载

饿汉式

饿汉式单例模式代码中，static变量会在类装载时初始化，此时也不会涉及多个线程访问该对象的问题，虚拟机保证只会装载一次该类，肯定不会发生并发访问问题，因此可以省略synchronized关键字。缺点是太过粗暴，类加载的原因有很多，无形中产生很多无用对象浪费内存。

问题：如果只是加载本类，而不是要调用getInstance()，甚至永远没有调用，则会造成资源浪费，不能懒加载

懒汉式

懒汉式在类加载后如果我们一直没调用getInstance()方法，就不会实例化对象，节约内存，实现了延迟加载，但是因为方法上加上synchronized，每次调用方法都会同步，所以对性能影响比较大。

双重检测锁模式

懒汉模式的改良版，也是属于使用时候才创建，但是与懒汉式相比大大提高了效率，并不是每一个请求都会阻塞，只有创建对象时才会导致线程阻塞，锁的粒度更小，需要进行两次判断，第一次判断是否需要创建对象，不需要直接返回存在的实例，避免不必要的阻塞和释放锁的系统资源浪费，提高效率；第二次判断主要针对场景是当发生锁的竞争时通过这个条件防止其他线程获得锁后再进行对象的创建。

有一点需要注意的是volatile关键字，它除了保证对象创建后对其他线程的可见性外还有一个重要作用是防止暴露没有初始完全的实例对象，volatile禁止指令重排序，确保instance = new SingleInstance()对应的指令不会重排序，但是因为对volatile的操作都在main memory中，而main memory是被所有线程共享，这里的代价是牺牲了性能，无法利用寄存器或者cache。

静态内部类模式

这种单例模式是线程安全的，内部类的static变量在类加载时被初始化，保证了内存中只能有一个这样的实例存在，而且只能被赋值一次，它可以实现用的时候再创建，主要利用了静态内部类和外部类之间没有很大的关联性，可以将他们当做两个类处理，对外部类加载时内部类不会被加载，当需要创建对象（调用静态方法时），内部类才会被加载。兼备了并发高效调用和延迟加载的优势。
